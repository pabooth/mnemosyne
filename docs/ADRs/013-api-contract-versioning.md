## Context

ADR-012 settled mnemo-core as the single required deployed service, with mnemo-ui, mnemo-curator, and mnemo-bot as optional services that consume its API, alongside first-party clients (Obsidian plugin, Chrome extension) and Agentic access via MCP. mnemo-curator and mnemo-bot do not yet exist. Building them against an API that is still being discovered organically during mnemo-core implementation risks rework on both sides every time the contract shifts.

mnemo-core already has an informal REST boundary (`mnemo_core/api/`) with preview/ingest endpoints and governance constraints enforced there (ingest is the only path permitted to publish; preview must remain side-effect free). The next build phase adds further functionality — the on-demand Indexer trigger, index reconciliation, and whatever mnemo-curator needs to pull from mnemo-core — all of which will land on this same boundary.

A decision is needed on (a) whether the API contract is designed before implementation or discovered alongside it, and (b) how the API is versioned so that future breaking changes don't destabilise clients that don't exist yet but will.

---

## Considered Options

### Design approach

#### Option 1: Schema-as-you-build

Design Pydantic models and routes as each piece of functionality is implemented; Swagger falls out as documentation rather than as an upfront gate.

**Pros:**
- No separate design phase before implementation begins
- Contract evolves naturally alongside the pipeline logic it serves

**Cons:**
- mnemo-curator and mnemo-bot have no stable target to build against until mnemo-core's implementation settles
- Contract changes discovered mid-implementation risk rework on both sides of the boundary

#### Option 2: Contract-first via route stubs *(chosen)*

Define routes and Pydantic request/response models with no business logic behind them. FastAPI generates the OpenAPI/Swagger document from these stubs automatically; that generated document becomes the review artifact, iterated on and agreed before pipeline logic is implemented.

**Pros:**
- mnemo-curator and mnemo-bot get a stable, reviewed target to build against before they exist
- The "spec" is generated by FastAPI, not hand-authored — the stub-first approach is close to free
- Surfaces gaps in the endpoint set (e.g. index trigger, reconciliation, and future curator integration routes) before code is written

**Cons:**
- Requires a design pass before any pipeline logic can be written
- Stubs can go stale if implementation later diverges from the agreed shape without updating the ADR

### Versioning scheme

#### Option 1: Header-based versioning

e.g. `Accept: application/vnd.mnemo.v1+json`.

**Pros:**
- Keeps URLs clean

**Cons:**
- Invisible in Swagger UI's route listing
- Requires custom OpenAPI tooling to represent well

#### Option 2: Query param versioning

e.g. `?version=1`.

**Pros:**
- Simplest to implement

**Cons:**
- Easiest to omit by accident
- Tends to leak into caching and logging in awkward ways

#### Option 3: URI path versioning *(chosen)*

e.g. `/api/v1/ingest`, `/api/v1/process`, implemented as a distinct version router package mounted under `/api/v1` in the FastAPI app.

**Pros:**
- Visible directly in the generated Swagger doc's route listing
- Each version is an isolated router package that can be frozen once superseded
- Matches the governance boundary requirement: a version bump can never silently change what an existing client's `ingest` call does

**Cons:**
- Running multiple versions in parallel increases surface area once `v2` exists
- Some duplication across version router packages is expected

---

## Decision

1. mnemo-core's API surface is designed contract-first: routes and Pydantic schemas are stubbed before implementation, and the resulting generated OpenAPI document is the artifact reviewed and agreed before pipeline logic is built behind it.
2. The API is versioned by URI path. Each version lives in its own router package (e.g. `mnemo_core/api/v1/`) mounted under its own `/api/v{n}` prefix in the FastAPI app. Once a version is superseded, its router is frozen except for bug fixes — behavioural changes go into the next version.

---

## Justification

Contract-first wins because mnemo-curator and mnemo-bot don't exist yet and need something concrete to build against; discovering the contract alongside their implementation would mean both sides moving at once. FastAPI's automatic OpenAPI generation makes the stub-first pass cheap: stubbing a route and its Pydantic models produces the review artifact directly, with no separate spec-authoring step.

URI path versioning wins on visibility and isolation: it shows up directly in the generated Swagger doc rather than requiring header inspection, and mounting each version as its own router package means a version can be frozen and left alone once superseded — which matters specifically for the governance-boundary code in `pipeline/git.py` and `pipeline/publish.py`, where silently changing behaviour under an existing client is unacceptable. The trade-off — more surface area once multiple versions run in parallel — is accepted as the cost of that stability.

---

## Enforcement

- How the decision is upheld in practice: new endpoints are added by first defining the route signature and Pydantic models under the relevant version's router, with a stub response, before pipeline logic is written.
- Each API version is a self-contained router package (`mnemo_core/api/v{n}/`) mounted under `/api/v{n}`; a version is never modified to change behaviour once a later version exists.
- CodeRabbit path instructions for `mnemo-core/mnemo_core/api/**` continue to apply per-version and should be extended to flag any cross-version behavioural coupling.
- Any proposal to change this approach should reference and supersede this ADR.

---

## Consequences

Positive:
- mnemo-curator and mnemo-bot can be built against a reviewed, stable contract instead of one still being discovered.
- The Swagger doc becomes free, accurate documentation rather than a maintenance burden.
- Existing clients are insulated from breaking changes once a version is frozen.
- Version boundaries make the governance guarantees (ingest-only publish path, side-effect-free preview) easy to audit per version.

Negative / trade-offs:
- Running multiple versions in parallel increases surface area and maintenance cost once `v2` exists.
- Requires discipline not to patch `v1` semantics under time pressure rather than cutting a new version.
- Some duplication across version router packages is expected and accepted as the cost of stability.
